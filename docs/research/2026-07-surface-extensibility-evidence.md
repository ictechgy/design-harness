# Surface Extensibility Evidence (2026-07)

Method: multi-agent web evidence sweep run 2026-07-08 (six parallel sweeps: standards/regulation, mobile tooling, academic, capture bridges, competitor architecture, candidate surfaces; plus a completeness critic). Every claim below carries a source. Unresolved items are listed under Open Questions — they are stated as unknown, not assumed. This document backs ADR-002 and the ROADMAP "Surface horizons" section.

| PDF/UA-2 | ISO 14289-2:2024 + Matterhorn Protocol (31 checkpoints / 136 failure conditions); veraPDF v1.30.2 (2026-06-08) validates deterministically, open source | The only non-web surface with an official-testable standard AND an open deterministic validator | [ISO](https://www.iso.org/standard/82278.html), [veraPDF releases](https://github.com/veraPDF/veraPDF-library/releases) |

Policy-matrix consequence: on an app surface, deterministic `failure` could anchor only to WCAG 2.2 (via WCAG2ICT/WCAG2Mobile mapping) or EN 301 549 Annex C tests. Vendor HIG thresholds (Apple 44pt, Google 48dp) are platform guidance — `official-pattern`-tier (risk ceiling), not official-testable; a project pinning them in its own design-guide.yaml is the `project-contract` path. KS X 3253 items need individual triage (a minority machine-testable; the certification scheme itself is part human-judged).

## 2. Mobile tooling: capture is commoditizing, engines separate from front-ends

- Google ATF (Apache-2.0): exactly 14 `AccessibilityHierarchyCheck` classes over a normalized element tree + screenshot; consumed by four front-ends (Accessibility Scanner, Espresso, Compose 1.8 test rules, Roborazzi). Touch target 48dp (32dp edge/IME); contrast 4.5:1/3:1 sampled from screenshot pixels. Slow-moving but alive (4.1.1, 2024-03-29). [GitHub](https://github.com/google/Accessibility-Test-Framework-for-Android)
- ATF labels pixel-derived contrast results with dedicated **heuristic** result IDs (`RESULT_ID_TEXTVIEW_HEURISTIC_CONTRAST_NOT_SUFFICIENT`) vs declared-color results — external validation that result strength must key on evidence provenance. [TextContrastCheck.java](https://github.com/google/Accessibility-Test-Framework-for-Android/blob/master/src/main/java/com/google/android/apps/common/testing/accessibility/framework/checks/TextContrastCheck.java)
- Apple: `performAccessibilityAudit` (Xcode 15/iOS 17, WWDC23-10035) — typed audit categories (contrast, hitRegion 44pt, dynamicType, textClipped, elementDescription, traits); same engine as Accessibility Inspector; requires a booted simulator. [Apple docs](https://developer.apple.com/documentation/xcuiautomation/xcuiaccessibilityaudittype)
- Deque axe DevTools Mobile does **not** run axe-core: purpose-built rules over native accessibility trees, mapped to WCAG + WCAG2ICT; distributed via manual app / embedded SDK / Appium plugin sharing one ruleset; retrofitted computer vision onto Android contrast (2024) — commercial admission that pixel contrast needed hedging. [Product page](https://www.deque.com/axe/devtools/mobile-accessibility/)
- Evinced: five capture SDKs feeding one analysis engine; all require server-validated API key — the dependency class this repo forbids (differentiation argument). Ships a Mobile MCP Server (mobile a11y evidence for AI agents is being commercialized). [Evinced docs](https://developer.evinced.com/sdks-for-mobile-apps/intro/)
- Commodity capture seams a future adapter would target, never own: Appium (page-source XML + screenshot; iOS `snapshotMaxDepth` default 60 silently truncates deep trees — capture-integrity must be schema-visible), Maestro (accessibility-layer hierarchy dump; cli-2.6.1, 2026-06-12), Roborazzi (JVM render + ATF checks, no emulator — Android-only asymmetry; iOS always needs a simulator). [Appium settings](https://appium.github.io/appium-xcuitest-driver/4.19/settings/), [Roborazzi a11y](https://github.com/takahirom/roborazzi/blob/main/roborazzi-accessibility-check/README.md)
- Automated coverage consensus: ~20–40% of a11y issues; Google ships Scanner output as "suggestions" with an explicit no-guarantee disclaimer — convergent with our copy guardrails. [Scanner FAQ](https://support.google.com/accessibility/android/faq/6376582?hl=en)

## 3. Academic evidence: families transfer, calibration does not

- Miniukovich & De Angeli (CHI 2015): same eight visual metrics explain ≤49% variance on web but ≤32% on iPhone apps — metrics transfer in kind, lose a third of explanatory power. [ACM](https://dl.acm.org/doi/10.1145/2702123.2702575)
- UEyes (CHI 2023): eye-movement patterns and saliency-model performance differ across webpage/desktop/mobile/poster — surface is a first-class parameter of every perceptual heuristic. [ACM](https://dl.acm.org/doi/10.1145/3544548.3581096)
- Our check families already exist on native with measured precision: MotorEase (ICSE 2024) touch-target etc. ~90% acc / <9% FP; AccessiText (FSE 2022) text scaling 87.6% precision; OwlEyes (ASE 2020) display defects 85% precision from screenshots alone; Groundhog (ASE 2022) crawler 86% precision. They encode 44pt/48dp — not WCAG's 24 CSS px. [MotorEase](https://arxiv.org/abs/2403.13690), [SoK survey](https://arxiv.org/html/2411.19727v1)
- Native structural trees are unreliable in ways the web DOM is not: CLAY (CHI 2022) — raw Android view hierarchies routinely contain invalid objects; Apple built Screen Recognition (CHI 2021) and its TOCHI 2024 report generator around **pixels** because app a11y metadata is often missing/wrong. Pixel and tree channels diverge on native (they coincide on web). [CLAY](https://arxiv.org/abs/2201.04100), [Screen Recognition](https://dl.acm.org/doi/10.1145/3411764.3445186)
- Quality judgment stays immature everywhere: UI foundation models (ScreenAI, Ferret-UI 2, UI-TARS) do grounding/navigation, not quality; UIClip (UIST 2024) is the only purpose-built quality scorer at ~74% pairwise designer agreement; WebDevJudge (2025-10): best LLM judge 66.06% vs 84.82% human-human — and **no study measures web→mobile judge-reliability transfer**. Judge = needs-review, re-validated per surface. [UIClip](https://dl.acm.org/doi/10.1145/3654777.3676408), [WebDevJudge](https://arxiv.org/abs/2510.18560)
- Touch-target thresholds are surface-specific by construction: WCAG 2.2 SC 2.5.8 = 24 CSS px (AA); Apple HIG 44pt; Material 48dp; empirical thumb studies ~9-10mm (MobileHCI 2006). The check transfers; the threshold and unit system do not. [W3C Understanding 2.5.8](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html), [Parhi et al.](https://dl.acm.org/doi/10.1145/1152215.1152260)
- Mobile ground truth for "is it good?" is scarce: Rico has no quality labels; Enrico only a binary good/bad curation filter; UIClip's BetterApp = 1.2K ratings. Mobile detectors cover 31/55 known a11y issue types (SoK 2024). [Rico](https://dl.acm.org/doi/10.1145/3126594.3126651), [Enrico](https://dl.acm.org/doi/10.1145/3406324.3410710)

## 4. Capture bridges, cost-ordered

| Bridge | Status | What survives / what breaks | Source |
| --- | --- | --- | --- |
| Storybook story iframes (`/iframe.html?id=…`) | **Works today**, zero adapter, inside local-HTTP policy | Full DOM/ARIA; page-scoped checks (page-lang, density) misfire on bare component iframes → needs a component-mode check profile, not capture code | [Storybook test-runner docs](https://storybook.js.org/docs/writing-tests/integrations/test-runner) |
| RN Web / Expo web | **Works today** via dev server or static export | Real DOM + ARIA (role→HTML element inference); all check families + Korean copy checks valid. Audits the web projection, NOT the native binary ("subtle rendering differences" per Expo) — claims must be scoped | [RNW a11y](https://necolas.github.io/react-native-web/docs/accessibility/), [Expo web](https://docs.expo.dev/workflow/web/) |
| Playwright device emulation | Config flag away | Valid: responsive layout math, meta-viewport, touch flags, DPR. | Flutter web | Partial: canvas-only rendering since 3.29 (HTML renderer removed) | Screenshots OK; semantics tree (`flt-semantics` ARIA DOM) exists ONLY if `ensureSemantics()` is activated — audit precondition; ALL computed-style checks produce garbage on canvas (keep-all/line-break, font metrics, CSS contrast) | [Flutter web a11y docs](https://docs.flutter.dev/ui/accessibility/web-accessibility), [#163199](https://github.com/flutter/flutter/issues/163199) |

| Native app evidence ingest (Appium/adb/XCUITest/Maestro) | Adapter = XML/tree→measurements mapping (days); device loop = ongoing flake cost (never own it) | Derivable: rendered text inventory (Korean copy checks incl. placeholder-leak, parser-free josa stay valid), roles/names, bounds (target size), screenshot. ## 5. Competitor architecture: the pattern that survives

- Deque: rule **code** does not cross surfaces even for the market leader — axe-core (web), axe DevTools Mobile, axe-windows are separate engines; what crosses is the criteria/standards mapping (WCAG + WCAG2ICT). [axe mobile](https://www.deque.com/axe/devtools/mobile-accessibility/), [axe-windows](https://github.com/microsoft/axe-windows/blob/main/docs/solution.md)
- Applitools / Percy / Chromatic: adapters only capture-and-serialize one evidence artifact (DOM snapshot / archive); one core renders/judges. Percy consolidated adapter sprawl onto @percy/cli. [Percy core](https://www.npmjs.com/package/@percy/core?activeTab=readme), [Ultrafast Grid](https://applitools.com/docs/topics/overview/using-the-ultrafast-grid.html)
- Chromatic (funded team) stayed Storybook-only ~7 years until 2024-03-27 before adding surfaces 2–3, reusing the same core — the strongest "demand-gated expansion" precedent. [Chromatic 3](https://www.chromatic.com/blog/chromatic-3/)
- ESLint went language-agnostic one language at a time (JSON 2024-10 → CSS 2025-02 → HTML 2025-05), only after the plugin API shipped. textlint (mostly one maintainer, a decade): zero bundled rules, frozen TxtAST contract, pluggable parsers — the solo-maintainer existence proof. Stylelint documents the honesty pattern: "no guarantee that core rules work with custom syntaxes." [ESLint blog](https://eslint.org/blog/2024/10/eslint-json-markdown-support/), [textlint](https://github.com/textlint/textlint), [Stylelint syntaxes](https://stylelint.io/developer-guide/syntaxes/)
- Appium 2.0 (GA 2023-07) unbundled every platform driver from core — drivers version independently against a frozen protocol. [Appium Pro](https://appiumpro.com/editions/122-installing-appium-20-and-the-driver-and-plugins-cli)
- Deaths: PhantomCSS archived 2018 (its capture engine died underneath it); BBC Wraith archived 2026-01-16; web-eval-agent — our own category — archived 2026-06-27 after ~1 year. Capture-coupled tools die with their capture layer. [PhantomCSS](https://github.com/HuddleEng/PhantomCSS), [web-eval-agent](https://github.com/Operative-Sh/web-eval-agent)
- DTCG tokens 2025.10 (first stable, 2025-10-28; Community Group report, not standards-track) + Style Dictionary v5: a versioned **data schema** is what crosses platforms, and even the flagship implementation lags the spec — schema contracts outlive implementations. [DTCG announcement](https://www.w3.org/community/design-tokens/2025/10/28/design-tokens-specification-reaches-first-stable-version/)

## 6. Candidate surfaces, ranked

Reachable today (contract holds, zero capture code): Storybook/component pages (Storybook 9/10 ships axe natively — our value there is the non-axe layer: heuristic design checks, Korean copy, epistemic tiering); design-system docs/marketing pages; PWA as plain web (Lighthouse 12 removed the PWA category May 2024 — evidence-against PWA-specific checks); in-page data-viz (input side; checks would be a new heuristic family via Chartability — 50 heuristics, EuroVis 2022, CC-BY-SA 3.0 share-alike license diligence required, heuristic/needs-review tier only).

Adapter-needed, demand-gated: Electron/WebView2 (local CDP; smallest widening); native mobile apps (evidence-ingest format, not owned capture; Android-first per the Roborazzi/ATF JVM asymmetry); email templates; PDF via veraPDF delegation.

Email deserves an honest note — two sweeps returned opposing recommendations. Against: a Chromium render is structurally non-representative of email clients (systematic false negatives — trust-killing in the inverse direction from false positives). For: EMC 2026 (376,348 emails) found 99.88% fail automated a11y checks, and the top three failure classes (missing `dir` 97.41%, missing `lang` 95.66%, layout tables without `role=presentation` 83.78%) are deterministic DOM checks we already ship or could trivially add; Korean copy checks transfer wholesale (rendered, post-interpolation), and Korea's dominant newsletter platform (Stibee) has no Litmus-class QA ecosystem. Resolution recorded in ROADMAP horizons: the honest subset is copy + semantic checks with claims scoped to the template source ("never claims about client rendering"); client-compat = versioned caniemail data at heuristic tier; screenshot farms out of scope. Gated on owner decision — neither current goal centers email.

Evidence-against (do not build): owned mobile capture / device farm; Windows UIA / macOS AXAPI desktop-native surface (two extra per-OS toolchains, near-zero check reuse); Tauri-on-macOS; slide decks (no open deterministic checker to delegate to); PWA-specific checks; email client screenshot farm.
