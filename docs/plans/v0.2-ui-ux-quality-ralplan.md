# RALPLAN: v0.2 Research-Backed UI/UX Quality

Date: 2026-07-04

This plan applies the research in [UI/UX Quality Basis](../research/ui-ux-quality-basis.md) to Design Harness.

## Goal

Turn UI/UX research into source-backed, evidence-bearing criteria that deterministic checks, reports, fixtures, and agent prompts can all reference.

## Recommended Architecture

```text
criterion registry
  -> check runtime
  -> evidence ledger
  -> audit schema
  -> report renderer
  -> agent adapter / fix prompt
```

## Key Decisions

- Criteria become first-class objects.
- Findings carry `criterionId`, `sourceRefs`, `determinism`, `resultKind`, `observed`, `expected`, and `humanReviewRecommended`.
- Deterministic, heuristic, and subjective outputs are separated in JSON and Markdown reports.
- Only official/testable sources can support deterministic pass/fail language.
- Pattern, heuristic, emerging research, and philosophical sources support risks, recommendations, or optional critique.
- v0.2 remains an evidence-grounded audit assistant, not an automated design-quality judge.

## Determinism Matrix

| Determinism | Result kind | Report wording |
| --- | --- | --- |
| `deterministic` | `failure` | Fails a scoped, implemented criterion check |
| `deterministic` | `risk` | Measurable risk found in the captured evidence |
| `heuristic` | `risk` | Likely UX risk; human review recommended |
| `heuristic` | `needs-review` | Suggestive evidence only |
| `subjective` | `needs-review` | Optional critique, never an audit failure |

Disallowed:

- `subjective` + `failure`
- `heuristic` + `failure` unless a deterministic sub-check proves the issue
- pass/fail language backed only by pattern libraries, emerging research, or philosophy

## Branch Plan

### 1. `feature/v0.2-criterion-registry`

Deliver:
- Criterion registry.
- Source registry or source metadata.
- Finding schema extensions.
- Evidence ledger fields.
- Report wording contract.
- 2-3 representative fixtures.

Gate:
- Existing v0.1 CLI outputs remain compatible.
- `audit.json` still validates.
- New schema fields are documented.
- Report language cannot overclaim conformance or objective quality.

### 2. `feature/v0.2-semantic-a11y-checks`

Deliver:
- Missing accessible names.
- Missing form labels.
- Missing informative image alt text.
- Heading sanity checks.
- Landmark sanity checks.
- Repeated ambiguous label risks.

Gate:
- Good/bad fixture pair per deterministic check.
- Findings include criterion, source, selector, viewport, observed/expected evidence.
- Ambiguous cases emit risk or needs-review, not failure.

### 3. `feature/v0.2-responsive-readability`

Deliver:
- Narrow viewport reflow check.
- Fixed-width risk.
- Sticky/fixed obstruction risk.
- Line length risk.
- Tap target risk where geometry supports it.

Gate:
- Checks are tagged by runtime type: static DOM, computed style, viewport sweep, interaction simulation, or human review.
- Noisy checks support confidence downgrades.
- Existing desktop/mobile audit remains ergonomic.

### 4. `feature/v0.2-interaction-state-checks`

Deliver:
- Form error association risks.
- Color-only state risks.
- Disabled control explanation risks.
- Dynamic status/live-region risks.
- Modal focus risk.
- Custom control semantics risk.
- Moving/autoplay content controls where feasible.

Gate:
- Static page-load limitations are explicit.
- Interaction-only checks require a scenario or are marked needs-review.

### 5. `feature/v0.2-report-source-links`

Deliver:
- Report sections grouped by deterministic failures, deterministic risks, heuristic review prompts, and optional subjective critique.
- Source-backed criteria appendix.
- Iteration prompt grouped by likely implementation area: semantics, layout, forms, interaction state, content, visual polish.

Gate:
- Top recommendations are evidence-linked.
- Report does not say "WCAG compliant", "accessible", "good design", "best practice violation", or "objectively better" without scoped qualification.

### 6. `feature/v0.2-docs-and-fixtures`

Deliver:
- Contributor guide for adding criteria and checks.
- Fixture catalog.
- Documentation of what the harness does and does not certify.
- Adapter workflow updates.

Gate:
- `pnpm build`, `pnpm typecheck`, `pnpm test`, `pnpm validate`, and smoke example pass.
- A contributor can add a new check by following one documented pattern.

## Review Loop

For each branch:

1. Implement the slice.
2. Run local verification.
3. Run code review.
4. Fix actionable findings.
5. Re-review until no material issues remain.
6. Merge only after gates pass.

## Risks And Controls

| Risk | Control |
| --- | --- |
| Criteria churn after implementation | Lock a small initial registry before check expansion |
| Subjective critique reads like a failure | Enforce determinism/result kind matrix |
| Source overreach | Use source strength and copy guardrails |
| Noisy heuristics reduce trust | Support confidence downgrades and future suppression/justification metadata |
| Dynamic checks overclaim static evidence | Classify checks by runtime type |
| v0.1 regression | Keep CLI output files and required fields compatible unless a documented schema migration is introduced |

## Definition Of Done

- Criteria are source-backed and first-class.
- Findings are traceable to criteria and evidence.
- Reports separate deterministic failures, risks, heuristic prompts, and subjective critique.
- Report copy avoids conformance and objective-quality overclaims.
- Deterministic checks have fixtures and tests.
- v0.1 golden path stays stable.
